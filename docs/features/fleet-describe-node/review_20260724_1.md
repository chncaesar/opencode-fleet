# Review 1: fleet-describe-node

Date: 20260724
Branch: main

## Summary

`fleet_describe_node` 的整体实现框架合理，ping → send prompt → 解析 JSON → 格式化输出的流程可以工作。但存在 **2 个 Critical 问题**（session 状态污染 + JSON 提取逻辑的结构性误判风险）和 **3 个 Major 问题**（capabilitySummary 逻辑缺陷、超时路径缺少 escalate_hint、write/edit 分析错误），以及若干 Minor 问题。共 **11 个问题**（2 Critical / 3 Major / 6 Minor）。

---

## Issues

### Critical

- [C1] **session 状态污染：`send()` 会创建并绑定一个新 session，覆盖节点上可能已存在的工作 session** — `src/tools.ts:1072`

  `handleDescribeNode` 通过 `ctx.sessions.send(node, prompt, { cwd, agent: "build" })` 发送诊断 prompt，而 `SessionManager.send()` 的行为是：若当前节点没有绑定 session 则自动创建并绑定，若已有绑定则复用。这带来两个问题：

  1. **节点已有工作 session 时**：诊断 prompt 被追加到该 session 的消息历史中，污染了正在运行的任务上下文。
  2. **节点没有 session 时**：`send()` 创建一个新 session 并将其绑定为"当前 session"，之后 master 对该节点的所有 `fleet_send_message` 调用都会进入这个诊断 session，而不是用户期望的干净上下文。

  设计文档要求 `fleet_describe_node` 是"首次使用节点前调用"的能力探测工具，其执行不应改变节点的 session 状态。正确做法是：使用一次性 session（创建 → 发 prompt → 等待 → 不绑定到 `SessionManager`），或在发送后将 session binding 恢复原状。

- [C2] **JSON 提取用 `indexOf("{")` + `lastIndexOf("}")` 会在嵌套对象中产生结构性误判** — `src/tools.ts:1089-1096`

  此策略取的是整个 reply 中"第一个 `{`"到"最后一个 `}`"的子串。如果 slave 的回复是：

  ```
  Here is the config: {"permission": {"bash": {"*": "allow"}}}
  Note: see field "key" for details.
  ```

  `lastIndexOf("}")` 会停在第一个 `}}}` 的最后，这里还好。但若 slave 在 JSON 后面附加了含 `}` 的文本（如花括号内的示例或错误描述），`lastIndexOf` 会越界到外层，截取出非法 JSON，导致 `JSON.parse` 失败并静默降级为"无法解析"。更危险的是：如果 JSON 内部某个字段的值本身含 `}`（如 bash pattern `"if [[ x ]]; then { echo y; }"`），而 slave 在 JSON 外面附加了额外文字，`lastIndexOf` 会错误地截断 JSON 本身，使解析出的对象不完整但仍合法（partial JSON），导致**权限规则被静默丢弃**而不报错。

  正确做法是使用增量 JSON parser 或通过匹配嵌套深度找到完整 JSON 对象边界，而非假设"最后一个 `}` 就是 JSON 的结束"。

### Major

- [M1] **`capabilitySummary` 的 `write`/`edit` 分析逻辑有误：只要有任何 `allow` 规则就报告"allowed"，忽略了 `deny` 规则的存在** — `src/tools.ts:1024-1033`

  ```typescript
  const allAllowed =
    Object.values(write ?? {}).some((v) => v === "allow") ||
    Object.values(edit ?? {}).some((v) => v === "allow");
  if (allAllowed) {
    notes.push("file writes/edits: allowed (some patterns)");
  } else {
    notes.push("file writes/edits: may require approval");
  }
  ```

  这段代码：
  - 变量名是 `allAllowed` 但实际检测的是"是否存在任意一条 allow 规则"，语义自相矛盾。
  - 完全不报告 `deny` 规则，如 `"write": { "*": "deny", "/tmp/*": "allow" }` 会被报告为"allowed"，master 可能因此向敏感路径发送写请求。
  - 不报告 `ask` 规则，master 无法知道哪些写操作会触发 approval 阻塞。

  该函数的目的是帮助 master 建立正确心智模型，但错误的摘要比没有摘要更危险。

- [M2] **超时路径缺少 `escalate_hint`，与设计文档的人工介入触发机制不一致** — `src/tools.ts:1073-1078`

  设计文档（§ "人工介入触发条件"）规定工具返回值应附加 `escalate_hint: true`，但当前 `ToolResult` 类型是纯文本（`content: Array<{type:"text"; text:string}>`），没有结构化字段。`fleet_send_message` 的超时路径同样缺少此字段（已在其他 review 中记录），但 `fleet_describe_node` 的超时更值得关注：如果诊断工具本身就超时，master 对该节点的能力一无所知，此时应明确提示升级给人工而不只是说"try fleet_get_session_status"。

  当前超时返回：
  ```
  return err(`Node "${nodeName}" did not respond ... try fleet_get_session_status.`)
  ```
  这是一个无结构的纯文本错误，缺少对 master 的决策引导（应该等？重试几次后升级？）。

- [M3] **工具 description 没有说明"会占用/创建 session"，master 无法预判副作用** — `src/tools.ts:330-353`

  工具 description 只说"Sends a diagnostic prompt to the node"，但没有告知 master：
  - 此调用**可能创建一个新 session**（若节点无 session）
  - 此调用**会污染现有 session**（若节点已有 session）
  - 正确的调用时机是"首次使用节点前"（设计文档原意）

  master AI 读取 description 来决定何时以及是否安全调用此工具。缺少副作用说明会导致 master 在任务执行中途调用 `fleet_describe_node` 做"探查"，从而破坏正在进行的会话。

### Minor

- [m1] **`capabilitySummary` 没有分析 `read`/`glob`/`search` 等工具的权限，输出不完整** — `src/tools.ts:981-1035`

  只分析了 `bash`、`write`、`edit` 三个工具，但 opencode permission 支持对任意工具（包括 `read`、`glob`、`task`、`webfetch` 等）设置规则。如果用户配置了对读工具的 deny（如沙箱环境），master 不会从摘要中得知，可能反复发出被拒绝的读请求。

- [m2] **`capabilitySummary` 没有检测 `write` 和 `edit` 均无规则的情形** — `src/tools.ts:1024`

  当 `!write && !edit` 时，函数完全跳过文件写权限的说明，notes 中不会有任何关于写操作的条目，master 得不到任何提示（既不知道是 allow 也不知道是 ask/deny）。

- [m3] **`formatPermissions` 对 bash pattern 的对齐用 `padEnd(30)` 硬编码，超过 30 字符的 pattern 会破坏列对齐** — `src/tools.ts:970`

  opencode 的 bash pattern 支持完整 glob 语法（如 `"git -C /home/zjc/projects/* push *"`），超过 30 字符时 `padEnd` 无效，输出列对齐错乱。

- [m4] **超时错误消息中的反引号语法 `'opencode debug config'` 与其他地方的代码引用风格不一致，且对 master AI 可读性无影响** — `src/tools.ts:1075-1077`

  次要风格问题，不影响功能。

- [m5] **`formatPermissions` 空 permission 节的消息有语义错误：空 `{}` 并不等于"所有操作 default to ask"** — `src/tools.ts:960-962`

  opencode 的默认行为取决于启动参数（`--auto` 下 `ask` 自动通过），不是固定的"ask"。空 permission section 在 `--auto` 模式下等同于"所有操作自动允许"，这与消息中的描述相反，可能误导 master 认为会有 approval 阻塞。

- [m6] **`PermissionRules` 类型定义将 verdict 声明为 `string` 而非 `"allow" | "deny" | "ask"`** — `src/tools.ts:942`

  ```typescript
  type PermissionRules = Record<string, Record<string, string>>;
  ```

  verdict 应为 `"allow" | "deny" | "ask"`，使用 `string` 导致编译器无法检查 `capabilitySummary` 中 `v === "allow"` 等比较的穷举性，也无法在值不合法时给出编译期警告。

## Escalate

- [ESCALATE] **实现选择了"通过 slave session 执行 `opencode debug config`"而非设计文档首选的"SSH 文件读取"方案**。

  设计文档列出两种读取方式：SSH 文件读取 vs opencode debug config。当前实现选择了后者，存在以下非技术性权衡需要用户决策：

  1. **SSH 文件读取** 需要 fleet MCP 有 SSH 访问权限（凭证、host key）；但读取 managed config（`/etc/opencode/`）是幂等操作，不创建 session，不污染节点状态。
  2. **通过 slave session 执行** 不需要额外的 SSH 配置，但带来了 [C1] 的 session 污染问题，且 `opencode debug config` 是否总是返回 permission 字段（而不是只返回合并后的 config 而 permission 为空）需要实际验证。

  这个选择影响到工具的部署要求和可靠性，建议用户在 [C1] 修复方案中一并决定：是引入 SSH 文件读取，还是坚持 session 方案并接受其限制。
